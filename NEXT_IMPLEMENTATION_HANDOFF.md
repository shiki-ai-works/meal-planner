## 目的

このファイルは、次に「ソフトの中身」を本格的に実用化するための引き継ぎメモです。

今のアプリは、ポートフォリオとして見せるための README、スクリーンショット、公開導線 E2E、画像出典チェックが整っている。次は、実ユーザーが登録して、迷わず使い始め、翌日も続きから使える状態にする。

たとえるなら、展示用の模型から、実際に水道と電気が通った家へ進める段階です。

## 実装したい 8 項目

### 1. 本番データでの実運用確認

Supabase 本番 project に migration（データベース変更履歴）を適用し、実際のユーザー登録、ログイン、保存、再読み込みが通るか確認する。

やること:

- 本番 Supabase の URL / anon key / service role key を Vercel 環境変数に設定する。
- `supabase/migrations` を番号順に本番 DB へ適用する。
- seed recipe、recipe detail、recipe image URL が本番 DB に入っているか確認する。
- RLS を確認する。RLS は Row Level Security の略で、ユーザーごとに見える行を制御するデータベース側の権限設定。
- 新規ユーザーで、設定、献立、在庫、買い物リストが保存されるか手動確認する。

完了条件:

- 新規ユーザーが signup から入れる。
- ログアウト後に再ログインしても、保存済みデータが残っている。
- 他ユーザーのデータが見えない。

### 2. 認証後の E2E

現在の E2E は公開ページ中心。次は、ログイン後の実利用ルートを自動検査する。

E2E は end-to-end の略で、ユーザーの入口から出口までをまとめて確認する検査。

やること:

- テスト用ユーザーを用意する。
- signup または login から入る。
- 初期設定を保存する。
- 献立生成を実行する。
- 1 食を固定する。
- 在庫を追加する。
- 買い物リストに反映されることを確認する。
- ログアウト、再ログイン後も状態が残ることを確認する。

完了条件:

- `npm run release:check` とは別に、認証付き E2E を実行できる。
- 本番または staging の URL に対しても同じ検査が走る。
- テスト用 credentials（認証情報）は `.env.local` や Vercel env に置き、repo へ commit しない。

### 3. 初回ユーザー導線

初めて来た人が、何を入力すれば献立が出るのか迷わないようにする。

やること:

- 初回 setup で、人数、苦手食材、アレルギー、自炊頻度、目標を聞く。
- 入力後すぐに最初の 1 週間の献立を生成できるようにする。
- 未設定ユーザーは dashboard ではなく setup へ自然に誘導する。
- setup 完了後は dashboard へ戻す。

完了条件:

- 新規ユーザーが 3 分以内に最初の献立を見られる。
- 空の dashboard に放り出されない。
- あとで settings から変更できる。

### 4. レシピ品質と量

実用アプリとしては、35 件の recipe だけだと献立の変化が少ない。使い続けるなら、選択肢の厚みが必要。

やること:

- レシピを 80〜120 件程度へ増やす。
- 朝食、昼食、夕食、弁当、時短、節約、高たんぱく、作り置き、子ども向けなどの軸を持たせる。
- 材料、手順、PFC、カロリー、画像 URL、画像出典をそろえる。
- 追加画像は author / license / source page URL を必ず控える。

完了条件:

- 1 週間の献立を複数回生成しても、同じ料理ばかりにならない。
- `recipe-images:sources-check:strict` が pass する。
- レシピ詳細で、材料と手順が実際に料理できる粒度になっている。

### 5. 献立生成の納得感

ユーザーが「なぜこの献立になったのか」を理解できるようにする。献立生成を、占いではなく小さな参謀に近づける。

やること:

- 生成結果に理由を持たせる。
- 例: 「在庫の卵を使うため」「昼が重いので夜は軽め」「固定枠を避けて選択」。
- 生成ロジック側で、採用理由や除外理由を作る。
- UI では短い理由だけ表示し、詳細は必要な時に開ける形にする。

完了条件:

- 献立カードに短い理由が出る。
- 固定枠、外食枠、在庫、苦手食材の影響が説明できる。
- 生成理由が空や嘘にならない。

### 6. エラー・空状態・読み込み状態

実用ソフトは、うまくいった時より、うまくいかなかった時に信頼を失いやすい。ここを丁寧にする。

やること:

- データがない時の empty state を用意する。empty state は空状態の案内表示。
- 通信中の loading state を用意する。loading state は処理中の表示。
- 保存失敗、ログイン期限切れ、画像読み込み失敗の error state を用意する。error state は失敗時の表示。
- 再試行ボタンや setup への誘導を置く。

完了条件:

- 主要ページで真っ白画面にならない。
- 失敗時にユーザーが次に何をすればよいか分かる。
- 通信が遅くても操作中であることが伝わる。

### 7. スマホでの実使用感

献立アプリは、台所や買い物中にスマホで使われる。PC で美しくても、片手で使いにくいと実用性が落ちる。

やること:

- dashboard、recipe detail、shopping list、inventory、settings を mobile viewport で確認する。
- 買い物リストのチェック操作を片手で押しやすくする。
- sticky action を検討する。sticky action は画面下などに固定される主要操作ボタン。
- レシピ詳細の材料と手順が小さい画面でも読みやすいようにする。
- 横スクロールや文字のはみ出しを潰す。

完了条件:

- 375px 幅でも主要操作が破綻しない。
- 買い物リストを店内で素早くチェックできる。
- recipe detail が読みやすい。

### 8. データ削除・エクスポート

公開アプリとして、ユーザーが自分のデータを扱えるようにする。信頼の土台になる機能。

やること:

- ユーザーデータの export を追加する。export は外へ書き出すこと。
- JSON 形式または CSV 形式で、設定、在庫、献立、買い物リストを出せるようにする。
- ユーザーデータ削除を追加する。
- 削除前に確認 modal を出す。
- privacy policy と画面上の説明を合わせる。

完了条件:

- ユーザーが自分のデータを取得できる。
- ユーザーが自分のデータを削除できる。
- 削除後にログインしても、消したデータが復活しない。

## 推奨順序

1. 本番 Supabase と保存系の手動確認
2. 認証後 E2E の土台作り
3. 初回 setup 導線
4. 空状態、読み込み状態、エラー状態
5. スマホ実使用感
6. 献立生成理由
7. レシピ拡充
8. データ export / delete

理由は、まず「保存できるか」を固めないと、その上に何を積んでも崩れるから。土台、導線、安心感、便利さの順で進めるのがよい。

## 検査方針

作業前後で、最低限これを確認する。

```powershell
npm.cmd run check
npm.cmd run release:check
```

認証付き E2E を追加したら、専用 script を作る。

```powershell
npm.cmd run e2e:auth
```

まだ `e2e:auth` は存在しない。作るなら、テスト用ユーザーの扱いを先に決める。

## 注意

- `.env.local` の秘密情報は commit しない。
- Supabase の service role key はブラウザ側へ出さない。
- 画像を増やす時は、出典、作者、license を必ず残す。
- ユーザーデータ削除は慎重に実装する。最初は staging で試す。
- 既存の `release:check` を壊さない。
